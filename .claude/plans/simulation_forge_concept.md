## 1. Executive Summary: The Democratization of the Fracture

The `metaverse.center` platform is not merely a game; it is an act of collective, adversarial writing. The existing Shards (*Speranza*, *Cité des Dames*, *Spengbab's Whore House*) were not generated by throwing generic keywords at an LLM. They were painstakingly crafted using a rigorous literary architecture, constrained by specific philosophical frameworks (Baudrillard, Fisher, Calvino), and mechanically grounded in the "Bleed" vectors.

**The Simulation Forge** is the user-facing web application designed to democratize this creation process—without sacrificing its intellectual or mechanical rigor. It allows authenticated "Cartographers" to generate new Shards.

This document details the logical architecture, the deeply embedded literary constraints, and the **Clean Technical Design** optimized for our existing stack and low-resource cloud environments (e.g., Railway Free/Hobby plans).

## 2. Design Philosophy: Constraint as Creativity

### 2.1 The Literary Architecture of the Forge
Drawing from our foundational document (`docs/concept.md`), the Forge must embed the following literary devices directly into its UX:

1.  **The Major Arcana (Taxonomy of Catastrophe):** The UI steps are mapped to the Tarot. World geography falls under *The Tower* (structure, sudden ruin). The Bleed falls under *The Lovers* (connection, contamination).
2.  **The Vectors (Channels of Contamination):** Users *must* select how their world bleeds into others (Memory, Commerce, Language, Violence, etc.).
3.  **The Philosophical Anchor:** Every Shard must ground its lore in a real-world literary/cultural theory proposed by the AI during the research phase.

### 2.2 The Systems Architecture of the Forge
1.  **Iterative Refinement Loop:** Chunked generation allowing users to "lock" good entities and regenerate bad ones.
2.  **Strict Schema Enforcement:** Utilizing **Pydantic AI** for function-calling, ensuring the LLM output matches our PostgreSQL schemas exactly.
3.  **Sequential Materialization:** To operate within a 512MB RAM ceiling, all heavy asset processing is handled one-by-one, never in parallel.

## 3. The User Journey: A Four-Phase Descent

### Phase I: The Astrolabe (Conceptualization & Deep Research)
*Arcanum: The Fool*
Uses an AI Agent (LangGraph + Tavily) to scrape the web for themes related to the user's "Seed." The user selects a **Philosophical Anchor**.

### Phase II: The Drafting Table (Structural Expansion)
*Arcanum: The Emperor & The Tower*
The AI generates the City name, Zones (with `characteristics` tags), Streets (with optional descriptions), Agents, and Buildings. Users can manually edit text or trigger micro-mutations. Each entity type uses chunk-specific prompts with word-count guidance (200-300 words for agent character/background, 150-250 words for building descriptions), geographic context (city/zone names), and diversity instructions (varied genders, factions, building conditions). Output quality is enforced via `Field(description=...)` on Pydantic models, ensuring LLM output matches the downstream image-generation templates.

### Phase III: The Darkroom (Aesthetic Calibration)
*Arcanum: The Moon*
A split-pane view for tuning the "Master Style Prompt." Users perform limited "Test Exposures" (single image renders) to calibrate the visual output.

### Phase IV: The Ignition (Materialization)
*Arcanum: Judgement*
The final batch execution. The backend moves the draft into production tables and starts the background asset pipeline.

## 4. Technical Architecture: Deep Dive

To maintain the project's **Clean Architecture**, the Forge will be implemented as a first-class module within the existing backend structure, strictly avoiding "hacks" or temporary workarounds.

### 4.1 Reusable Backend Components
We will leverage and extend the following existing services to ensure zero logic duplication:
*   **`GenerationService` (Reused):** We will utilize the existing `generate_portrait_description` and `generate_building_description` methods. The Forge will simply act as a new "Client" of this service.
*   **`ImageService` (Reused & Extended):** The Forge will reuse the robust `_upload_dual_resolution` and `_convert_to_avif` logic. We will add a new `materialize_assets` method to handle the batch processing required for a full Shard.
*   **`SimulationService` (Utility Reuse):** We will reuse the `_slugify` and count-enrichment logic to ensure Forge-generated Shards behave identically to pre-authored ones.

### 4.2 The Draft State Machine (`forge_drafts`)
Following the project's database pattern, we introduce a staging table. This table will be managed by a new `ForgeDraftService` which, like `SimulationService`, handles global (non-simulation-scoped) entity logic.

```sql
CREATE TABLE public.forge_drafts (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id uuid REFERENCES auth.users(id) NOT NULL,
    current_phase text NOT NULL DEFAULT 'astrolabe', 
    seed_prompt text,
    philosophical_anchor jsonb, 
    taxonomies jsonb DEFAULT '{}'::jsonb,
    geography jsonb DEFAULT '{}'::jsonb,
    agents jsonb DEFAULT '[]'::jsonb,
    buildings jsonb DEFAULT '[]'::jsonb,
    ai_settings jsonb DEFAULT '{}'::jsonb, 
    status text DEFAULT 'draft', 
    created_at timestamptz DEFAULT now(),
    updated_at timestamptz DEFAULT now()
);
```

### 4.3 The Orchestrator (Pydantic AI)
The `ForgeOrchestratorService` will be built using **Pydantic AI**, aligning with our existing commitment to Pydantic v2.
*   **Strict Typing:** We will use the project's existing Pydantic models (from `backend/models/forge.py`) as the `output_type` for the LLM. Each field carries a `Field(description=...)` with word-count guidance and image-generation hints to ensure LLM output quality matches the DB prompt templates used in the established generation pipeline.
*   **Typed Geography:** `ForgeGeographyDraft` uses typed sub-models (`ForgeZoneDraft`, `ForgeStreetDraft`) rather than `list[dict]`, ensuring zones include `characteristics` tags and streets include optional `description` fields.
*   **Rich Prompts:** `_build_chunk_prompt()` builds chunk-type-specific prompts that include the user's seed prompt, anchor context, geographic context (city/zone names for agent/building chunks), word-count guidance matching DB templates, image-ready hints, and diversity instructions.
*   **Agentic Search:** The `ResearchService` will use **LangGraph** to manage the iterative state of the "Astrolabe" search process, storing its graph state in PostgreSQL to remain memory-efficient for Railway.

### 4.4 Background Asset Pipeline (FastAPI `BackgroundTasks`)
In alignment with our "Lean Infrastructure" mandate, we will replace the existing synchronous script pattern (e.g., `generate_spengbab_images.py`) with FastAPI's native `BackgroundTasks`.
*   **Sequential Loop:** To stay within the **512MB RAM** ceiling, the background worker will process images one-by-one. 
*   **Thread Safety:** CPU-intensive image conversions will be offloaded to `anyio.to_thread.run_sync` to prevent blocking the main event loop, maintaining API responsiveness for other players.

### 4.5 Database Layer (Sovereign Logic)
Critical business rules are pushed to the database to ensure integrity:
*   **`fn_materialize_shard`:** A PL/pgSQL function that executes the move from `forge_drafts` to production tables in a single atomic transaction. This prevents "orphaned" simulation components. Python should only *call* this function, never manage the multi-table insert loop itself.
*   **Trigger-Based Quotas:** Economic barriers (token deduction) are enforced via PostgreSQL triggers on `forge_drafts` (BEFORE UPDATE), ensuring they cannot be bypassed by client-side or API-level bugs.
*   **Taxonomy Integrity:** Reuses existing validation triggers to ensure `primary_profession` and `building_type` exist in the `simulation_taxonomies` for that Shard.

### 4.6 Frontend: The `ForgeStateManager`
The frontend will continue to use the **Preact Signals** pattern established in `AppStateManager.ts`, providing a reactive, low-overhead bridge between the Forge UI and the Supabase Realtime progress updates.

### 6.4 CSS Architecture: Token-First Pattern
To maintain visual consistency and performance, all Forge components will strictly follow the project's **Shadow DOM styling pattern**:
*   **Token Adoption:** Every color, spacing value, and animation curve MUST be derived from the CSS custom properties in `styles/tokens/` (e.g., `var(--space-4)`, `var(--color-primary)`, `var(--ease-dramatic)`).
*   **Shared Style Injection:** Rather than monolithic stylesheets, components will import and compose small, functional CSS modules from `components/shared/` (e.g., `static styles = [formStyles, gridLayoutStyles, css\`...\`]`).
*   **Theming Parity:** By using the standard tokens, the Forge interface will automatically inherit the active simulation's theme (Brutalist, Cyberpunk, etc.), maintaining the "Integrated" look.

## 7. Comparative Analysis of Generation Strategies

### Strategy A: The "God Prompt" (Rejected)
*   **Concept:** The user types one massive paragraph. The AI extracts the entire database structure, lore, and aesthetic parameters in one monolithic API call.
*   **The Flaw:** LLMs suffer from "lost in the middle" syndrome. If asked to generate 15 distinct JSON objects (Agents, Buildings, Zones) in one go, it *will* hallucinate schemas, skip fields, or degrade in narrative quality.

### Strategy B: The "Node Graph" (Too Complex)
*   **Concept:** A visual, Unreal Engine-style node editor where users manually drag connections between themes, vectors, and character archetypes.
*   **The Flaw:** Massively increases the barrier to entry. We want writers and worldbuilders, not technical artists.

### Strategy C: The "Progressive Refinement Wizard" (Approved)
*   **Concept:** The 4-phase, chunked approach detailed in this document.
*   **The Strength:** It guarantees structural integrity (the DB schemas) while maximizing creative agency. By separating the Text generation from the Image generation, we prevent catastrophic token waste.

## 12. Implementation Retrospective & Known Weaknesses

This section documents the actual implementation process, the specific technical hurdles encountered, and the resulting architectural decisions. It serves as a guide for future maintainers.

### 12.1 The Pydantic AI Integration Hurdle (Version 1.66.0)
*   **The Goal:** We intended to use Pydantic AI for strict JSON schema enforcement when querying OpenRouter (which proxies to `claude-3.5-sonnet`).
*   **The Problem:** Pydantic AI's documentation and API surface have evolved rapidly. Initial attempts to initialize the `OpenAIModel` class with a custom `base_url` or `http_client` (to redirect to OpenRouter) resulted in `TypeError: OpenAIChatModel.__init__() got an unexpected keyword argument...`. 
*   **The Solution:** The correct pattern for Pydantic AI `v1.66.0` when using a third-party OpenAI-compatible provider (like OpenRouter) is to explicitly import `OpenAIProvider` and inject it into the `OpenAIModel`:
    ```python
    from pydantic_ai.providers.openai import OpenAIProvider
    from pydantic_ai.models.openai import OpenAIModel

    provider = OpenAIProvider(base_url="https://openrouter.ai/api/v1", api_key=key)
    model = OpenAIModel("anthropic/claude-3.5-sonnet", provider=provider)
    ```
*   **Weakness/Risk:** Pydantic AI is a relatively new and fast-moving library. This specific initialization pattern might be deprecated or changed in future versions (e.g., `v2.x`). Future upgrades to this library must be tested rigorously against the Forge's specific `OpenRouter` routing logic.

### 12.2 Routing Shadowing and FastAPI Prefixes
*   **The Problem:** During testing, `POST /api/v1/forge/drafts` repeatedly returned `405 Method Not Allowed`. 
*   **The Cause:** Two distinct issues contributed to this:
    1.  **Prefix Stacking:** `backend/routers/forge.py` defined `prefix="/api/v1/forge"`, and `app.include_router()` in `app.py` mistakenly added another `prefix="/api/v1"`, resulting in a hidden `/api/v1/api/v1/forge` route.
    2.  **SPA Catch-All Shadowing:** The `app.py` contained a global `serve_spa` catch-all route (`@app.get("/{full_path:path}")`) designed to serve the Vue/Lit frontend. Because the `forge` router wasn't loaded early enough, or the SPA route was too greedy, API requests were falling into the SPA handler.
*   **The Solution:** 
    *   Standardized the router to handle its own prefix: `router = APIRouter(prefix="/api/v1/forge")`.
    *   Added a strict exclusion to the SPA catch-all: `if full_path.startswith("api/"): return JSONResponse(...)`.
    *   Moved the `app.include_router(forge.router)` higher in the middleware stack.

### 12.3 Local JWT Authentication (Supabase Bypass)
*   **The Problem:** Local API testing via `curl` failed with `401 Unauthorized: Session expired or invalid.` 
*   **The Cause:** The `get_supabase` dependency in `backend/dependencies.py` attempts to set the user's JWT session against the actual Supabase GoTrue server (`client.auth.set_session(user.access_token, "")`). A locally minted Python JWT—even if signed with the correct `SUPABASE_JWT_SECRET`—is rejected by the GoTrue API because it lacks a corresponding valid session record in the database.
*   **The Solution/Workaround:** We modified `get_supabase` to catch the `AuthApiError`. If the token is validly signed by our secret (verified via `_decode_jwt`), we manually inject the token into the PostgREST headers (`client.postgrest.auth(user.access_token)`) instead of requiring a full GoTrue session. 
*   **Weakness:** This is a local-dev-only necessity. In production, tokens are always generated by GoTrue, so this fallback is rarely hit. However, developers writing E2E tests must ensure they generate tokens with the correct `aud: "authenticated"` claim.

### 12.4 The `fn_materialize_shard` Transaction Model
*   **The Decision:** We moved the logic that transfers data from `forge_drafts` to the live `simulations`, `zones`, `agents`, and `buildings` tables entirely into PostgreSQL via the `fn_materialize_shard` stored procedure.
*   **Why it's Good:** It guarantees absolute atomicity. If the AI hallucinates a non-existent building type, the `simulation_taxonomies` foreign key constraint will fail, and the *entire* shard creation rolls back. The database remains perfectly clean.
*   **Weakness:** Complex JSON parsing in PL/pgSQL can be opaque to debug. If the frontend alters the schema of the `agents` or `geography` JSON in the future, the `fn_materialize_shard` function *must* be manually updated, creating a tight coupling between the frontend state shape and the database logic.

### 12.5 Required Environment Keys
For the Forge to function locally or in production, the following keys MUST be present in `.env`:
1.  **`OPENROUTER_API_KEY`**: (e.g., `sk-or-v1-...`) Required for Pydantic AI to generate the lore.
2.  **`REPLICATE_API_TOKEN`**: (e.g., `r8_...`) Required for Flux-Dev image generation.
3.  **`TAVILY_API_KEY`**: Required for the deep-web research phase. *(Note: The code has a fallback to generate 'mock' research if this is missing, to allow local dev without a paid Tavily account).*
4.  **`SETTINGS_ENCRYPTION_KEY`**: Required by the `backend.utils.encryption` module to decrypt user BYOK (Bring Your Own Key) entries.

## 13. Technical Reference: Comprehensive File & Logic Map

This section provides an exhaustive list of every file and technical pattern implemented for the Simulation Forge.

### 13.1 Database Schema (Supabase/Postgres)
*   **`public.forge_drafts`**: Staging table for WIP simulations. Uses `jsonb` for geography, agents, and buildings to allow for highly flexible AI-driven drafting without violating production schema constraints.
*   **`public.user_wallets`**: Stores `forge_tokens` and `is_architect` status.
*   **`fn_materialize_shard(uuid)`**: The most complex logic block. It performs a "Deep Insert" by parsing the draft JSON. Crucially, it creates a `v_zone_map` (JSONB) in-memory to map AI-generated zone names to their newly created UUIDs, allowing `city_streets` to maintain referential integrity.
*   **`trg_enforce_forge_quota`**: A safety trigger that locks the `ignition` phase unless a token is present.

### 13.2 Backend Service Architecture (FastAPI)
*   **`ForgeDraftService`**: Standard CRUD for drafts. Includes the **BYOK Encryption Logic** using `cryptography.fernet`.
*   **`ResearchService`**: Handles the "Astrolabe" phase. Uses `anyio.to_thread.run_sync` to wrap the synchronous Tavily search, preventing blocking of the async event loop.
*   **`ForgeOrchestratorService`**: The "Brain." It manages the transition between phases and coordinates the **Pydantic AI** (v1.66.0) agents. It reuses the project's core `ImageService` and `GenerationService`.
*   **`forge.py` (Router)**: Exposes the API at `/api/v1/forge`. It utilizes the new `require_architect()` dependency.

### 13.3 Frontend State & Component Tree (Lit)
*   **`ForgeStateManager.ts`**: The central source of truth using `@preact/signals-core`. 
    *   *Pattern:* Implements strict **Effect Disposal** in `connectedCallback`/`disconnectedCallback` to prevent memory leaks during wizard navigation.
*   **`VelgForgeWizard.ts`**: The parent orchestrator. Uses a sub-render pattern (`_renderCurrentPhase`) to switch between the 4 Arcanum views.
*   **`VelgForgeTable.ts`**: Manages the drafting grid. 
    *   *Logic:* Implements **Optimistic Updates**—when a user edits an agent card, the signal updates locally before the debounced save hits the API.
*   **`VelgGameCard.ts` (Extended)**: Added `editable` property and the `description-change` custom event. This event uses the `bubbles: true, composed: true` flags to cross the Shadow DOM boundary.
*   **`AdminUsersTab.ts` (Extended)**: Now fetches the `user_wallet` via the updated `admin_list_users` RPC and provides a "Forge Access" sub-panel.

### 13.4 Security & Authentication Logic
*   **`SupabaseAuthService.ts`**: Updated the `_syncAppState` method to be `async`. It now calls `forgeApi.getWallet()` on every login/session-refresh to ensure the `isArchitect` signal is always accurate.
*   **`AppStateManager.ts`**: Added `isArchitect` (signal) and `canForge` (computed). The `canForge` property is the platform's primary UI gate.

### 13.5 Internationalization (i18n) Pattern
*   **Dynamic Placeholders:** Every AI-generated string is treated as a variable, but the "Labels" and "Instructions" are strictly wrapped in `msg()`.
*   **Status Indicators:** Progress messages like "Calibrating Rete..." and "Igniting Fracture..." are localized to ensure the Forge feels native to both English and German users.

### 13.6 Infrastructure & Performance Safety
*   **Sequential Backgrounding:** The batch generation loop in `ForgeOrchestratorService` uses a strict `for` loop (no `gather`) to ensure the **65MB AVIF conversion overhead** is never multiplied, keeping the Railway container stable under the 512MB limit.
